# Design: Tool Layer Enhancement

## Context

当前工具层已实现基础的工具注册、执行、超时和重试机制，但存在以下问题：

1. **缺少熔断保护**: 外部 API 工具（如搜索）不稳定时，重复调用会导致资源浪费和用户体验下降
2. **缺乏分类体系**: 所有工具扁平管理，无法按业务类型（内部/外部/系统）分类管理和查询
3. **参数定义格式**: 使用 Zod 而非行业标准的 JSONSchema，与外部工具调用格式不兼容

## Goals / Non-Goals

**Goals:**

- 实现熔断器组件，支持 CLOSED/OPEN/HALF_OPEN 三种状态
- 建立工具分类体系，支持5种分类：内部工具、外部API、文件系统、向量检索、代码执行沙箱
- 扩展 BaseTool 接口支持 JSONSchema 参数定义
- 保持向后兼容，不破坏现有工具实现

**Non-Goals:**

- 不实现复杂的工具编排流程
- 不修改现有的 planner/executor 核心逻辑
- 不引入新的外部依赖（使用原生实现）

## Decisions

### D1: 熔断器实现方式

**决定**: 实现状态机模式的熔断器，不使用第三方库

**理由**:

- 项目已有超时和重试机制，状态机模式可无缝集成
- 避免引入新依赖，保持轻量
- 完全可控，适合业务场景

**替代方案考虑**:

- `opossum` - 功能完善但增加依赖
- ` promises-concurrent` - 更复杂，不适合当前场景

### D2: 工具分类策略

**决定**: 在 BaseTool 中添加可选的 `category` 字段，支持多分类

**理由**:

- 一个工具可能属于多个分类（如"搜索API"同时是外部API和搜索类）
- 配置驱动分类，便于扩展新分类
- 保持简单，避免过度设计

### D3: JSONSchema vs Zod

**决定**: 支持双格式，Zod 优先，JSONSchema 可选

**理由**:

- 保持向后兼容，现有 Zod 定义无需修改
- 新工具可选择 JSONSchema
- 可通过工具函数互相转换

### D4: 熔断器粒度

**决定**: 以工具名为粒度，每个工具独立熔断

**理由**:

- 外部 API 工具通常独立，无共享资源
- 实现简单，维护成本低
- 与现有重试机制不冲突

## Risks\*\*[Risk / Trade-offs

] 熔断器状态持久化\*\*

- 问题: 进程重启后熔断器状态丢失
- 缓解: 当前仅内存实现，如需持久化可后续扩展

**[Risk] 分类查询性能**

- 问题: 工具数量增长后线性查询变慢
- 缓解: 使用 Map 缓存分类索引，O(1) 查询

**[Risk] 向后兼容性**

- 问题: 新增字段可能与现有工具冲突
- 缓解: 新增字段均为可选，使用鸭式类型判断

## Migration Plan

1. **Phase 1**: 扩展 BaseTool 接口，添加 category、timeout、retryConfig 可选字段
2. **Phase 2**: 实现 CircuitBreaker 类
3. **Phase 3**: 扩展 ToolRegistry，支持分类查询和熔断器集成
4. **Phase 4**: 更新 executor 集成熔断器
5. **Phase 5**: 更新文档和示例

## Open Questions

已确定：

- ✅ 熔断器指标暴露（监控）- 需要，通过 console.log 打印信息形式呈现
- ✅ 工具分类层级结构 - 不需要，保持扁平结构
- ✅ 分类权限控制 - 暂时不做，在 `ToolCategory` 类型中预留 `requiredRoles` 字段供将来扩展

### D5: 熔断器监控指标

**决定**: 通过 console.log 打印熔断器状态变化和执行统计

**理由**:

- 简单直接，无需额外监控基础设施
- 便于调试和问题排查
- 可对接日志收集系统

**监控内容**:

- 状态转换事件 (CLOSED → OPEN → HALF_OPEN → CLOSED)
- 执行统计 (成功次数、失败次数、拒绝次数)
- 当前状态和剩余恢复时间

### D6: 分类权限控制预留

**决定**: 在 `ToolCategory` 类型中预留 `roles` 字段，暂不实现权限检查逻辑

**理由**:

- 为将来功能扩展留出接口
- 不增加当前实现复杂度
- 便于后续接入认证系统
