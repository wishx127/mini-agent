## 背景

### 现状

现有的 Agent 执行模型遵循单次执行模式：`execute()` 调用一次规划器，执行返回的工具调用，然后立即返回。这种设计适合简单、直接的任务，但在以下场景中效果不佳：

- **工具失败恢复**：工具失败时没有重试或策略调整机制
- **多步推理**：复杂任务需要多轮"规划-执行-反思"迭代不被支持
- **动态上下文更新**：执行过程中新增信息无法用于调整策略
- **Token 管理**：累积对话历史和工具结果无法有效剪枝，导致 token 爆炸

### 核心问题

1. **无反馈循环**：工具结果没有系统评估，无法根据结果决策下一步
2. **单体执行**：所有工具调用一次性执行，不支持串行依赖或条件分支
3. **内存开销**：对话历史无限增长，没有摘要压缩机制
4. **工具重复调用**：无去重机制，规划器可能用相同输入多次调用同一工具
5. **终止条件模糊**：仅依赖规划器的布尔标志，缺乏语义理解

### 相关方

- **开发者**：需要清晰的状态机和内存接口便于调试和扩展
- **用户**：期望 Agent 从失败中优雅恢复，处理复杂任务
- **运维**：需要指标监控 Agent 性能，识别失败模式

---

## 目标 / 非目标

### 目标

1. **启用多轮执行**：建立显式的 OBSERVE → PLAN → ACT → REFLECT 反馈循环，支持任意迭代深度
2. **实现分层记忆系统**：分离工作记忆（近期上下文）、工具记忆（去重）、摘要记忆（压缩），防止 token 爆炸
3. **支持智能规划**：规划器支持输出多步计划并附加置信度；支持计划分支和条件执行
4. **增强失败恢复**：实现根因分析（工具错误 vs 规划器错误）和自动重试/降级策略
5. **语义终止条件**：从布尔标志升级到多维判断（信息增长、迭代预算、token 限制）
6. **维持 API 兼容性**：外部 Agent 接口不变，仅内部执行流程升级
7. **支持未来扩展**：为多 agent 分解、树形思维和 LangGraph 集成奠定基础

### 非目标

1. **LLM 模型选择**：不实现 LLM 切换逻辑（模型选择与架构正交）
2. **完整多 agent 系统**：本次重构是*为了*多 agent，不实现多 agent；单个 agent 保持单线程
3. **提示词工程**：不修改提示词构造方式（规划器接口签名可演进，但提示词逻辑可调优）
4. **流式执行**：不支持工具结果的实时流式推送（保持批量执行模型）

## 决策

### 决策 1：显式状态机 vs 递归调用

**选择**：在 `execute()` 内部实现显式状态机，包含 OBSERVE → PLAN → ACT → REFLECT 阶段和循环-直到-终止逻辑。

**理由**：
- **状态流清晰**：循环进度可视化，便于插入新阶段（如 VERIFY、RECOVER）
- **易于检测和日志**：每个阶段可记录结构化数据，指标收集直接
- **控制流透明**：循环终止条件显式，无隐藏递归深度
- **可测性强**：每个阶段可独立单元测试，状态转移可验证

**考虑的替选方案**：
- 递归规划器调用：实现简单但状态追踪困难，可观测性差
- 生成器模式：代码优雅但开发者不熟悉，调试困难

---

### 决策 2：三层记忆架构

**选择**：分离为三层：
1. **工作记忆**：最近 N 轮对话（规划器上下文）
2. **工具记忆**：所有工具调用含输入/输出/状态（去重、遥测）
3. **摘要记忆**：周期性摘要早期轮次（token 压缩）

**理由**：
- **职责分离**：每层有明确目标
- **Token 高效**：工作记忆紧凑（仅近期），摘要记忆压缩旧数据
- **去重便宜**：工具记忆使得检查近期相同工具调用低成本
- **多 agent 友好**：摘要记忆可跨 agent 共享，工作记忆保持本地

**考虑的替选方案**：
- 单历史+剪枝：难以判断删除什么，易失去重要信息
- 嵌入压缩：复杂度高，增加 LLM 调用，延后优化
- 树形历史：对当前需求过度复杂

---

### 决策 3：toolMemory 条目格式

**选择**：每条记录包含：
```json
{
  "tool": "string",
  "input": "object (JSON序列化)",
  "output": "string (超过500字符时截断)",
  "status": "success | failed | timeout",
  "timestamp": "ISO 8601",
  "iterationNum": "number"
}
```

**理由**：
- **去重键**：工具名 + 输入哈希快速查询
- **状态字段**：区分成功结果和失败，规划器可决定是否重试
- **有界输出**：防止工具记忆无限增长
- **时间戳+轮号**：支持按新近性过滤，便于清理策略

---

### 决策 4：规划器输出格式（多步计划）

**选择**：规划器返回结构化输出：
```json
{
  "type": "tool" | "final",
  "steps": [
    {"tool": "name", "input": {...}, "dependencies": []},
    {"tool": "name2", "input": {...}, "dependencies": [0]}
  ],
  "confidence": 0.75,
  "reasoning": "string (可选)"
}
```

**理由**：
- **多步支持**：规划器可建议串行或并行执行
- **依赖关系**：支持条件分支（"仅当工具A成功时调用工具B"）
- **置信度**：反思阶段用此判断是否信任计划或请求重新规划
- **推理**：便于调试规划器决策，支持未来解释性功能

**考虑的替选方案**：
- 每轮单工具调用：更简单但消除并行和依赖建模能力
- LangGraph DAG 风格：功能强大但过早复杂化，当前格式可映射到 DAG

---

### 决策 5：summaryMemory 生成策略

**选择**：在以下任一条件满足时触发摘要生成：
- **轮数触发**：每 N 轮（如 5 轮）生成一次，或
- **Token 触发**：工作记忆+近期工具结果超过阈值（如 8K tokens），或
- **语义触发**：信息密度下降时（重复工具结果表示探索完成）

使用 LLM 将轮次压缩为 1-2 句摘要，捕捉关键事实和决策。

**理由**：
- **混合触发**：防止无界增长和过早压缩
- **LLM 压缩**：语义压缩优于简单启发式，可提取关键信息
- **可调参数**：轮数 N、token 阈值可基于指标调优

**考虑的替选方案**：
- 固定滑动窗口：丧失历史，难以回放决策
- 无损压缩：节省空间但破坏可审计性
- 数据库存档：增加基础设施复杂度，延后

---

### 决策 6：语义终止条件

**选择**：满足以下任一条件则停止循环：
1. **规划器信号**：规划器输出 `type: "final"`
2. **无信息增长**：最近 N 个工具调用未返回新数据（收敛检测）
3. **最大迭代**：硬限制（默认 10 轮）
4. **Token 超预算**：对话历史+待处理工作超过阈值
5. **超时**：总执行时间超过限制（默认 60s）
6. **降级触发**：工具失败超过重试预算

**理由**：
- **多维判断**：无单一完美指标，组合多个信号提高鲁棒性
- **收敛检测**：防止陷入死路探索
- **硬限制**：保证可预测性，防止失控执行
- **降级逃生**：执行卡住时优雅退化

---

### 决策 7：反思阶段决策逻辑

**选择**：执行后，反思阶段评估：
1. **成功检查**：所有工具成功？是否超时？
2. **效用检查**：结果推进任务（新信息 vs 重复模式）？
3. **错误归因**：失败归因于工具/规划器/系统/token？
4. **下一步**：从以下选择：
   - `retry`：修改参数或换工具重新规划
   - `new_plan`：用更新上下文重新规划
   - `finalize_answer`：用当前知识生成答案
   - `fallback`：终止并使用降级策略

**理由**：
- **显式错误处理**：区分可恢复和不可恢复失败
- **自适应策略**：非所有失败都值得重试，部分表示任务不可行
- **置信度驱动**：用规划器的置信度决定重试概率

## 风险 / 权衡

| 风险 | 缓解策略 |
|------|---------|
| **Token 消耗增加（短期）** | 尽早实现 summaryMemory，测量影响，调优触发条件。接受临时增加作为可调试性的代价。 |
| **调试复杂度** | 在每个阶段转移处添加结构化日志。提供追踪回放工具用于事后分析。 |
| **规划器调用开销** | 多步规划减少多轮的总规划器调用（更少调用，上下文更好）。测试并优化热路径。 |
| **状态机膨胀** | 保持状态转移最小化，用阶段特定处理器避免 `execute()` 函数过大。 |
| **收敛检测误判** | 无信息增长检测可被规划器虚假摘要欺骗。在反思阶段添加语义验证层。 |
| **工具去重漏检** | 输入哈希可能错过语义相同但文字不同的调用。接受漏检，优先防止误检。 |

## 落地计划

### 第一阶段：基础（第 1 周）
- 创建新的 `ExecutionEngine` 类，包含 OBSERVE/PLAN/ACT/REFLECT 结构
- 实现 `ConversationHistory`、`ToolMemory`、`SummaryMemory` 类
- 添加状态机转移的测试套件
- **发布**：仅内部开发，无行为变化

### 第二阶段：集成（第 2 周）
- 更新规划器支持多步计划和置信度评分
- 将新记忆层集成到规划器上下文
- 实现反思阶段决策逻辑
- **发布**：灰度发布 5% 的 Agent 执行到新引擎

### 第三阶段：验证（第 3 周）
- 收集迭代深度、终止原因、工具成功率指标
- A/B 测试：代表工作负载上的旧 vs 新引擎
- 基于数据调整参数（最大迭代、token 阈值、摘要触发）
- **发布**：如果指标改进，扩展到 50% 流量

### 第四阶段：切换（第 4 周）
- 完成迁移到新引擎，删除旧单次执行代码
- 更新文档和内部 API
- 生产环境监控回归
- **发布**：100% 流量

### 回滚策略
- 旧 `execute()` 保留在独立模块，灰度路由可在引擎间切换
- 完整历史记录在 conversationHistory 中可用于回放/调试
- 指标仪表板展示旧 vs 新引擎性能，回归时易于切回

## 开放问题

1. **summaryMemory 触发调优**：从轮数触发（每 5 轮）还是 token 触发（每 8K tokens）开始？需要生产数据决策。
2. **收敛检测阈值**：检测到多少重复工具结果才判定"无新信息"？建议从 2 个连续相同结果开始。
3. **降级策略细节**：反思决策降级时，最终答案是什么？（当前最佳知识？错误消息？升级？）延后至能力 spec。
4. **多步计划执行**：依赖关系应严格（串行）还是软性（优选顺序）？为安全起见建议严格，后续可放宽。
5. **指标保留**：toolMemory 条目保留多长？建议 30 天内保留，更早的归档为摘要形式。

## 实现约束

- **语言**：TypeScript（现有代码库）
- **向后兼容**：外部 `Agent.execute()` 签名必须不变
- **测试**：ExecutionEngine 代码覆盖率 ≥80%
- **性能**：反思阶段完成时间 <100ms（避免反思中调用 LLM）
- **依赖**：不引入新外部库，用现有 LLM 集成生成 summaryMemory
