## 背景

### 现状

现有的 Agent 执行模型遵循单次执行模式：`execute()` 调用一次规划器，执行返回的工具调用，然后立即返回。这种设计适合简单、直接的任务，但在以下场景中效果不佳：

- **工具失败恢复**：工具失败时没有重试或策略调整机制
- **多步推理**：复杂任务需要多轮"规划-执行-评估-反思"迭代不被支持
- **动态上下文更新**：执行过程中新增信息无法用于调整策略
- **Token 管理**：累积对话历史和工具结果无法有效剪枝，导致 token 爆炸

### 核心问题

1. **无反馈循环**：工具结果没有系统评估，无法根据结果决策下一步
2. **单体执行**：所有工具调用一次性执行，不支持串行依赖或条件分支
3. **内存开销**：对话历史无限增长，没有摘要压缩机制
4. **工具重复调用**：无去重机制，规划器可能用相同输入多次调用同一工具
5. **终止条件模糊**：仅依赖规划器的布尔标志，缺乏语义理解

### 相关方

- **开发者**：需要清晰的状态机和内存接口便于调试和扩展
- **用户**：期望 Agent 从失败中优雅恢复，处理复杂任务
- **运维**：需要指标监控 Agent 性能，识别失败模式

---

## 目标 / 非目标

### 目标

1. **启用多轮执行**：建立显式的 OBSERVE → PLAN → ACT → EVALUATE → REFLECT 反馈循环，支持任意迭代深度
2. **实现分层记忆系统**：分离工作记忆（近期上下文）、工具记忆（去重）、摘要记忆（压缩），防止 token 爆炸
3. **支持智能规划**：规划器支持输出多步计划并附加置信度；支持计划分支和条件执行
4. **增强失败恢复**：实现根因分析（工具错误 vs 规划器错误）和自动重试/降级策略
5. **语义终止条件**：从布尔标志升级到多维判断（信息增长、迭代预算、token 限制）
6. **维持 API 兼容性**：外部 Agent 接口不变，仅内部执行流程升级
7. **支持未来扩展**：为多 agent 分解、树形思维和 LangGraph 集成奠定基础

### 非目标

1. **LLM 模型选择**：不实现 LLM 切换逻辑（模型选择与架构正交）
2. **完整多 agent 系统**：本次重构是*为了*多 agent，不实现多 agent；单个 agent 保持单线程
3. **提示词工程**：不修改提示词构造方式（规划器接口签名可演进，但提示词逻辑可调优）
4. **流式执行**：不支持工具结果的实时流式推送（保持批量执行模型）

## 决策

### 决策 1：显式状态机 vs 递归调用

**选择**：在 `execute()` 内部实现显式状态机，包含 OBSERVE → PLAN → ACT → EVALUATE → REFLECT 阶段和循环-直到-终止逻辑。

**理由**：

- **状态流清晰**：循环进度可视化，便于插入新阶段（如 VERIFY、RECOVER）
- **易于检测和日志**：每个阶段可记录结构化数据，指标收集直接
- **控制流透明**：循环终止条件显式，无隐藏递归深度
- **可测性强**：每个阶段可独立单元测试，状态转移可验证

**考虑的替选方案**：

- 递归规划器调用：实现简单但状态追踪困难，可观测性差
- 生成器模式：代码优雅但开发者不熟悉，调试困难

---

### 决策 2：三层记忆架构

**选择**：分离为三层：

1. **工作记忆**：最近 N 轮对话（规划器上下文）
2. **工具记忆**：所有工具调用含输入/输出/状态（去重、遥测）
3. **摘要记忆**：周期性摘要早期轮次（token 压缩）

**理由**：

- **职责分离**：每层有明确目标
- **Token 高效**：工作记忆紧凑（仅近期），摘要记忆压缩旧数据
- **去重便宜**：工具记忆使得检查近期相同工具调用低成本
- **多 agent 友好**：摘要记忆可跨 agent 共享，工作记忆保持本地

**考虑的替选方案**：

- 单历史+剪枝：难以判断删除什么，易失去重要信息
- 嵌入压缩：复杂度高，增加 LLM 调用，延后优化
- 树形历史：对当前需求过度复杂

---

### 决策 3：toolMemory 条目格式

**选择**：每条记录包含：

```json
{
  "toolCallId": "string",
  "toolName": "string",
  "arguments": "object (JSON序列化)",
  "result": "string (工具执行结果)",
  "status": "pending | success | failed | timeout",
  "error": "string (错误信息，可选)",
  "timestamp": "ISO 8601",
  "iteration": "number",
  "executionTime": "number (执行时间ms，可选)",
  "retryCount": "number (重试次数，可选)",
  "inputHash": "string (输入哈希，可选)"
}
```

**理由**：

- **输入哈希**：支持快速去重检查，无需深度比较对象
- **截断输出**：防止工具记忆无限增长，保留足够信息用于调试
- **迭代编号**：便于追踪工具调用发生在哪一轮循环

---

### 决策 4：并行执行波次模型

**选择**：将计划步骤按依赖关系分组为"波次"（wave），同波次步骤并行执行，波次间顺序执行。

**理由**：

- **最大化并行**：无依赖步骤同时执行，减少总耗时
- **依赖尊重**：确保数据依赖得到满足
- **可预测性**：波次模型易于理解和调试
- **资源控制**：可限制每波次的并发数

**考虑的替选方案**：

- 完全并行+等待：实现复杂，难以处理复杂依赖
- 完全串行：简单但低效

---

### 决策 5：反思阶段的多维度决策

**选择**：反思阶段基于以下维度综合决策：

- 工具成功率
- 信息增长率
- 规划器置信度
- 重试预算剩余

**理由**：

- **智能决策**：不依赖单一指标，综合评估
- **可配置**：不同风险偏好可调整权重
- **可解释**：决策理由清晰，便于调试

## 架构概览

```
┌─────────────────────────────────────────────────────────────┐
│                      ExecutionEngine                        │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐        │
│  │ OBSERVE │→ │  PLAN   │→ │   ACT   │→ │EVALUATE │        │
│  └─────────┘  └─────────┘  └─────────┘  └────┬────┘        │
│       ↑                                       │             │
│       └───────────────────────────────────────┘             │
│                                               ↓             │
│                                          ┌─────────┐        │
│                                          │ REFLECT │        │
│                                          └────┬────┘        │
│                                               │             │
└───────────────────────────────────────────────┼─────────────┘
                                                │
                       ┌────────────────────────┘
                       ↓
              ┌─────────────────┐
              │  done? ──Yes──→ │──→ 返回结果
              │       └──No──→  │──→ 继续循环
              └─────────────────┘
```

### 核心组件

1. **ExecutionEngine**：状态机主循环，协调各阶段执行
2. **StateDigestGenerator**：OBSERVE 阶段，生成状态摘要
3. **PlanningContextBuilder**：构建规划上下文
4. **ParallelExecutor**：ACT 阶段，并行执行工具
5. **DeltaDetector**：工具去重检测
6. **Evaluator**：EVALUATE 阶段，评估执行结果
7. **Reflector**：REFLECT 阶段，决策下一步
8. **TerminationChecker**：终止条件检查
9. **Memory**：分层记忆系统（工作记忆、工具记忆、摘要记忆）

## 风险与缓解

### 风险 1：循环无法终止

**缓解**：

- 硬限制：maxIterations、maxExecutionTime
- 软限制：token 预算、信息增长检测
- 预警机制：接近限制时通知

### 风险 2：Token 爆炸

**缓解**：

- 工作记忆大小限制
- 自动摘要生成
- Token 预算监控

### 风险 3：工具无限重试

**缓解**：

- 每工具重试预算
- 跨轮次累计计数
- 反思阶段决策

### 风险 4：状态机复杂度

**缓解**：

- 清晰的阶段接口
- 单元测试覆盖
- 结构化日志
