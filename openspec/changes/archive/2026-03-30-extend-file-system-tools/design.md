# Design: 文件系统工具扩展

## Context

当前文件操作工具集已实现以下功能：

- **ReadTool**: 读取文件内容，支持行范围选择和大小限制
- **GlobTool**: 文件模式匹配查找，支持深度控制和排序
- **GrepTool**: 正则表达式搜索文件内容，支持多文件搜索

这些工具覆盖了文件读取和查找场景，但缺少文件写入、创建、删除和移动能力。为了支持完整的代码生成和文件管理工作流，需要扩展工具集以支持：

- 目录结构遍历（LS）
- 文件创建（Create）
- 文件内容写入（Write）
- 文件删除（Delete）
- 文件移动/重命名（Move）

### 现有架构

文件操作工具基于以下架构：

- **BaseTool**: 所有工具的基类，提供统一的工具接口
- **ToolRegistry**: 工具注册和发现机制
- **PathValidator**: 路径安全验证，确保操作在项目目录内
- **ToolError**: 统一的错误处理机制

## Goals / Non-Goals

**Goals:**

- 实现与 Claude Code 类似的文件系统操作体验
- 保持与现有工具一致的设计模式和接口风格
- 确保所有写操作都有完善的安全验证
- 提供清晰的控制台反馈
- 危险操作（Delete）需要用户确认

**Non-Goals:**

- 不实现目录删除（避免误删整个目录树）
- 不实现文件权限修改
- 不实现软链接创建
- 不实现文件内容搜索替换（使用 Read + Write 组合实现）

## Decisions

### 1. 工具命名和参数设计

**决策**: 使用简洁的动词命名工具（ls, create, write, delete, move），参数使用 snake_case。

**理由**:

- 与现有工具（read, glob, grep）保持一致
- 清晰表达操作意图
- 符合 OpenAI Function Calling 的命名习惯
- 命名简洁，易于使用

**替代方案**: 使用更长的命名（如 list_directory, create_file），但简洁命名更符合现有风格。

### 2. 写入操作的安全策略

**决策**:

- 所有写入操作前必须验证路径在项目目录内
- 自动创建不存在的父目录
- Create 工具默认不覆盖已存在文件（`overwrite: false`）
- Write 工具默认覆盖已存在文件（`overwrite: true`）
- Move 工具默认不覆盖目标文件（`overwrite: false`）

**理由**:

- 防止意外写入到系统目录
- 简化文件创建流程
- 通过 `overwrite` 参数明确控制行为，避免误操作
- 不同工具的默认行为符合其语义（Create 强调新建，Write 强调写入，Move 强调安全移动）

### 3. Create 与 Write 的职责分离

**决策**:

- **Create**: 用于创建新文件，默认情况下如果文件已存在则返回错误，可通过 `overwrite: true` 强制覆盖
- **Write**: 用于写入内容，默认覆盖现有文件，可通过 `overwrite: false` 禁止覆盖
- **Move**: 用于移动/重命名文件，默认不覆盖目标文件

**理由**:

- Create 提供显式的新文件创建语义
- Write 提供灵活的内容写入，适用于大多数场景
- Move 提供完整的文件位置变更能力，避免 delete+write 的性能和元数据问题
- 三者互补，满足不同使用场景

### 4. Delete 工具的安全机制

**决策**:

- Delete 工具执行前必须获得用户手动确认
- 工具不直接执行删除，而是返回确认请求
- 用户授权后才执行实际删除操作
- 删除是永久的，无回收站功能

**理由**:

- 防止 Agent 误删重要文件
- 用户始终掌握最终控制权
- 符合危险操作的安全最佳实践

### 5. 错误处理策略

**决策**:

- 扩展 `FileOperationErrorCode` 枚举，新增写入、删除、移动相关错误码
- 所有错误包含详细的上下文信息（路径、操作类型等）

**新增错误码**:

- `WRITE_ERROR`: 写入操作失败
- `FILE_ALREADY_EXISTS`: 文件已存在（Create/Move 工具）
- `DELETE_ERROR`: 删除操作失败
- `MOVE_ERROR`: 移动操作失败
- `SOURCE_NOT_FOUND`: 源文件不存在（Move 工具）

### 6. 控制台输出设计

**决策**: 使用统一的图标和格式输出操作信息：

- LS: `📂 ls:`
- Create: `➕ create:`
- Write: `📝 write:`
- Delete: `🗑️ delete:`
- Move: `📦 move:`

**理由**:

- 与现有工具（📄 read, 📁 glob, 🔍 grep）保持一致
- 提供即时的操作反馈
- 便于用户跟踪 Agent 的文件操作

### 7. LS 工具的递归和排序

**决策**:

- 支持 `recursive` 参数进行递归遍历
- 支持 `sort_by` 参数按名称或时间排序
- 支持 `max_depth` 参数限制递归深度

**理由**:

- 递归目录查看是常见需求（如查看项目结构）
- 排序功能便于用户理解目录内容
- `max_depth` 防止递归过深导致的性能问题

### 8. Write 工具的大小限制

**决策**: 移除 1MB 大小限制

**理由**:

- 1MB 限制缺乏实际依据
- 过早优化是万恶之源
- 如果后续确实出现性能问题，可以再添加限制
- 保持与 Read 工具一致（Read 也没有大小限制）
