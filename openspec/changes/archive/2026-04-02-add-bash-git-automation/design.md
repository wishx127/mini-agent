## Context

项目目前已具备完整的文件操作能力，包括文件读写、编辑、搜索等。为了进一步提升自动化能力，需要添加 Bash 命令执行和 Git 版本控制操作功能。

当前技术栈：

- TypeScript/Node.js 运行时
- 基于插件的工具架构
- 现有的文件操作工具位于 `src/tools/plugins/file-operations/`
- 使用 `child_process` 模块进行进程管理

## Goals / Non-Goals

**Goals:**

- 实现基于人类监督的安全命令执行模式
- 实现完整的 Git 操作工具集（结构化参数 + execFile）
- 建立统一的命令安全机制（危险检测 + 人类确认）
- 实现进程树终止，确保资源清理
- 提供超时控制和输出限制
- 支持工作目录控制
- 实现审计日志记录
- 支持 Windows 和 Unix 跨平台

**Non-Goals:**

- 不支持交互式命令（需要用户输入的命令）
- 不支持长时间运行的守护进程
- 不实现完整的 Git GUI 功能
- 不提供 SSH 密钥管理功能（依赖 Git 原生配置）
- 不实现内存限制（Node.js 无跨平台方案）
- 不提供自动回滚机制（危险操作需要人类确认）

## Decisions

### 1. 安全模型：黑名单 + 人类确认

**决策**: 不使用白名单机制，改用黑名单检测 + 人类确认模式。

**理由**:

- 原白名单方案包含 node/python 等解释器，导致白名单形同虚设
- 黑名单 + 人类确认更灵活，同时保持安全性
- 人类可以正确评估复杂操作的真实风险
- 符合"人类监督执行模式"的定位

**黑名单检测**:

- `rm -rf /` - 危险的文件系统操作
- `mkfs`, `dd` - 磁盘操作
- Fork 炸弹等

**人类确认**:

- `git push --force` - 覆盖远程历史
- `git reset --hard` - 丢失未提交更改
- `git clean -fd` - 删除未跟踪文件
- 解释器的危险用法（如 `require('child_process')`）

### 2. Git 操作使用结构化参数 + execFile

**决策**: Git 操作使用结构化参数，通过 `child_process.execFile` 执行，不使用 shell。

**理由**:

- 避免 shell 注入攻击
- 参数以数组形式传递，不会被 shell 解释
- Git 命令语义清晰，适合结构化封装

**示例**:

```typescript
git_clone({
  url: 'https://github.com/user/repo.git',
  directory: 'repo',
});
```

### 3. 路径验证使用 realpath + 立即执行

**决策**: 使用 `fs.realpathSync.native()` 解析符号链接，验证后立即执行命令。

**理由**:

- 解析符号链接防止链接劫持
- 最小化验证到执行的时间窗口
- 接受残余的 TOCTOU 风险（人类监督模式下可接受）

### 4. 并发控制使用简单限制

**决策**: 最大 3 个并发，超限立即拒绝，不排队。

**理由**:

- 实现简单，无队列管理问题
- 调用方可自行决定重试策略
- 防止意外大量并发导致系统卡顿

### 5. 审计日志使用文件日志

**决策**: 使用 JSON 格式文件日志，保留 7 天。

**理由**:

- 简单，无外部依赖
- 按日期自动轮转
- 敏感信息脱敏处理
- 日志写入使用安全序列化，防止注入

### 6. 资源限制

**决策**: 实现超时和输出大小限制，不实现内存限制。

**理由**:

- Node.js 无跨平台内存限制方案
- 超时和输出限制已覆盖主要风险
- 内存限制作为未来增强项（Docker 模式）

**默认限制**:

- 一般操作超时：30 秒
- clone 操作超时：可配置（如 120 秒）
- 输出大小限制：100KB

### 7. 进程树终止

**决策**: 使用进程组信号确保子进程也被终止。

**理由**:

- npm install 等命令会创建大量子进程
- 只 kill 主进程会导致僵尸进程
- 需要先发 SIGTERM 等待退出，再发 SIGKILL

**Unix 实现**:

```typescript
execFile('kill', ['-TERM', `-${pid}`]);
await sleep(5000);
execFile('kill', ['-KILL', `-${pid}`]);
```

**Windows 实现**:

```typescript
execFile('taskkill', ['/PID', pid.toString(), '/T', '/F']);
```

### 8. Git 认证策略

**决策**: 不管理凭证，依赖 Git 原生配置。

**理由**:

- 不处理敏感信息，无泄露风险
- 简单，符合个人开发工具定位
- 用户已配置 SSH key 或 credential helper

### 9. 危险 Git 操作确认机制

**决策**: force push、hard reset、clean 等操作需要人类明确确认。

**理由**:

- 这些操作不可逆或影响其他协作成员
- 人类可以正确评估操作的影响
- AI 返回确认请求，由人类决定是否执行

**需要确认的操作**:
| 操作 | 风险 | 替代方案 |
|------|------|----------|
| git push --force | 覆盖远程历史 | 先 pull 再 push |
| git reset --hard | 丢失未提交更改 | git stash / reset --soft |
| git clean -fd | 删除未跟踪文件 | 手动备份 |

### 10. 跨平台兼容性

**决策**: 实现完整的 Windows 和 Unix 支持。

**理由**:

- 用户可能在不同平台使用此工具
- Windows 有不同的进程管理和路径处理

**需要适配**:

- Git 路径检测：Unix 用 `which`，Windows 用 `where`
- 进程终止：Unix 用 `kill`，Windows 用 `taskkill`
- 临时目录：使用 `os.tmpdir()`
- 路径分隔符：使用 Node.js path 模块处理

### 11. 错误处理策略

**决策**: 统一错误格式，包含错误码、类型、可重试性、确认标记。

**格式**:

```typescript
interface ToolError {
  code: string; // 如 'GIT_AUTH_FAILED'
  message: string; // 人类可读信息
  type:
    | 'SECURITY'
    | 'NETWORK'
    | 'GIT'
    | 'TIMEOUT'
    | 'RESOURCE'
    | 'CONFIRMATION';
  retryable: boolean; // 是否可重试
  requiresConfirmation?: boolean; // 是否需要人类确认
  details?: Record<string, any>;
}
```

### 12. 事务性和回滚

**决策**: 不提供自动回滚，操作前预检查，失败返回清晰错误信息。危险操作需要人类确认。

**理由**:

- 自动回滚可能干扰用户工作流
- Git 本身有 reflog 可手动恢复
- 人类确认机制已提供安全屏障

## Risks / Trade-offs

| 风险                 | 缓解措施                              | 残余风险             |
| -------------------- | ------------------------------------- | -------------------- |
| 命令注入攻击         | 使用 execFile（无 shell），结构化参数 | 低                   |
| 路径遍历攻击         | realpath 解析 + 项目目录验证          | 低                   |
| 资源耗尽             | 超时控制、输出限制、并发数限制        | 低                   |
| TOCTOU 竞态条件      | 最小化验证到执行的时间窗口            | 中（人类监督可发现） |
| Git 认证失败         | 依赖 Git 原生配置，提供清晰错误信息   | 低                   |
| 危险操作导致数据丢失 | 人类确认机制                          | 低                   |
| 僵尸进程             | 进程树终止机制                        | 低                   |
| 跨平台兼容性         | 平台检测和适配                        | 中（需要测试验证）   |

**Trade-offs**:

- 安全性 vs 灵活性: 采用黑名单 + 人类确认，平衡安全性和可用性
- 功能完整性 vs 简洁性: 优先实现常用功能，复杂操作留待后续
- 自动化 vs 可控性: 危险操作需要人类确认，让人类保持控制

## Migration Plan

1. **Phase 1**: 实现安全基础模块
   - 实现路径验证器（realpath + 项目目录检查）
   - 实现危险命令黑名单检测
   - 实现并发限制器（最大 3）
   - 实现审计日志模块
   - 实现进程树终止机制

2. **Phase 2**: 实现 Git 操作工具
   - 实现安全的 Git 执行器（execFile + 超时控制）
   - 实现 clone、status、commit、push 等核心操作
   - 实现危险操作确认机制
   - 统一错误处理格式

3. **Phase 3**: 实现 Bash 执行工具
   - 实现危险命令黑名单检测
   - 实现解释器危险用法检测
   - 集成安全模块和确认机制

4. **Phase 4**: 测试和文档
   - 编写单元测试和集成测试
   - 测试 Windows 和 Unix 跨平台
   - 更新工具注册表
   - 编写使用文档
