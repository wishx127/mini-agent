# 熔断器规范

## 新增需求

### 需求：熔断器状态管理

熔断器必须维护三种状态：CLOSED（正常）、OPEN（阻断）、HALF_OPEN（测试恢复）。

#### 场景：初始状态为 CLOSED

- **当** 创建新的 CircuitBreaker 实例时
- **则** 其状态必须为 CLOSED

#### 场景：达到失败阈值后转换为 OPEN

- **当** 在时间窗口内失败次数达到配置的阈值时
- **则** 熔断器必须转换到 OPEN 状态
- **且** 后续的工具执行必须立即被阻断，不尝试执行

#### 场景：超时后自动转换为 HALF_OPEN

- **当** 熔断器处于 OPEN 状态达到配置的时间后
- **则** 熔断器必须转换到 HALF_OPEN 状态
- **且** 允许有限的测试请求来检查服务是否恢复

#### 场景：成功时转换回 CLOSED

- **当** HALF_OPEN 状态收到成功响应时
- **则** 熔断器必须转换回 CLOSED 状态
- **且** 重置失败计数器

#### 场景：HALF_OPEN 状态失败时转换回 OPEN

- **当** HALF_OPEN 状态收到失败响应时
- **则** 熔断器必须转换回 OPEN 状态
- **且** 重置半开超时计时器

### 需求：执行保护

熔断器在 OPEN 状态下必须提供即时失败响应。

#### 场景：OPEN 状态时快速失败

- **当** 在熔断器处于 OPEN 状态时尝试执行工具时
- **则** 执行必须立即被拒绝，并抛出 CircuitOpenError
- **且** 拒绝不计入失败尝试

#### 场景：HALF_OPEN 状态时执行尝试

- **当** 在熔断器处于 HALF_OPEN 状态时尝试执行工具时
- **则** 执行应被允许
- **且** 结果（成功/失败）应触发状态转换

### 需求：配置选项

熔断器必须支持可配置的参数。

#### 场景：自定义阈值配置

- **当** 使用自定义 failureThreshold 创建 CircuitBreaker 时
- **则** 应使用配置的阈值而不是默认值

#### 场景：可配置的时间窗口

- **当** 使用自定义 resetTimeout 创建 CircuitBreaker 时
- **则** 熔断器应保持 OPEN 状态指定时间后再转换到 HALF_OPEN

### 需求：监控指标

熔断器必须通过日志打印暴露状态变化和执行统计。

#### 场景：状态转换时打印日志

- **当** 熔断器状态发生转换时
- **则** 必须通过 console.log 打印状态转换信息
- **且** 日志应包含：工具名称、转换前状态、转换后状态、时间戳

#### 场景：执行完成后打印统计

- **当** 工具执行完成时（成功或失败）
- **则** 应打印执行统计信息
- **且** 统计应包括：成功次数、失败次数、被拒绝次数

#### 场景：OPEN 状态时打印剩余恢复时间

- **当** 熔断器处于 OPEN 状态时
- **则** 应定期打印剩余恢复时间
- **且** 日志应包含工具名称和预计恢复时间点
